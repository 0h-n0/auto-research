# Reproducibility 7-tuple checklist

`research.lab.notebook` が POSTMORTEM §6 "Reproducing this failure" で **失敗を再現可能にする
最低保証** を担保する SoT。

> 科学研究の本質: 失敗そのものも再現できなければ、なぜ失敗したかの検証が出来ない。
> 成功 run の再現性だけでなく **失敗 run の再現性** が必須。

## 7-tuple

| # | 項目 | 保存先 | 既存 (v0.13.0) / 新規 (v0.14.0) | 不在時の挙動 |
|---|------|--------|------------------------------|------------|
| 1 | **Code rev** (git revision) | `events.jsonl` の `git_rev` フィールド | 既存 (PostToolUse hook) | error: code state 不明では再現不可、abort |
| 2 | **Config** | `06_RUNS/<id>/config.yaml` | 既存 | error: config なしでは再現不可、abort |
| 3 | **Dependencies** (uv.lock snapshot) | `06_RUNS/<id>/uv.lock` | **新規 (v0.14.0)** | warning: deps drift で 1 ヶ月後再現不能、reproduce.sh は best-effort |
| 4 | **Random seed** | `events.jsonl.seed` フィールド (or config 内) | 既存 | warning: 非決定論なら再現は確率的 |
| 5 | **Data version** | `data_lineage.md` の dataset hash | 既存 (v0.3.0+) | warning: dataset drift で input 不一致 |
| 6 | **Hardware** (GPU model + CUDA) | `events.jsonl.env` (gpu_model, cuda_version, os) | 既存 (要拡張確認) | warning: 異種 GPU で挙動異なる可能性 (特に numerical sensitivity) |
| 7 | **Reproduce command** | `06_RUNS/<id>/reproduce.sh` | **新規 (v0.14.0)** | error: 再現コマンドなしでは reproducibility 不成立、abort |

## reproduce.sh の必須要件

```bash
#!/bin/bash
# Auto-generated reproduce script for <run_id>
# Generated by research.lab.notebook v0.14.0
# Original status: <succeeded|failed> at <timestamp>
# Original code rev: <git_rev>

set -euo pipefail   # ← 必須 (silent fail は再現性破壊)

cd "$(dirname "$0")"

# Frozen deps (uv.lock pinned to <hash>)
uv sync --frozen --quiet

# Original command (extracted from events.jsonl)
exec uv run --frozen python <entry_script> --config config.yaml "$@"
```

### 必須の構文要素

1. **`set -euo pipefail`**: silent fail を禁止 (errexit / nounset / pipefail)
2. **`cd "$(dirname "$0")"`**: スクリプトのある dir に移動 (どこから呼んでも動く)
3. **`uv sync --frozen`**: uv.lock 凍結を強制 (deps drift 防止)
4. **`exec uv run --frozen python ...`**: 元コマンド (events.jsonl から復元)
5. **shebang**: `#!/bin/bash`

### Optional だが推奨

```bash
# Code rev check (実行前に warning だけ、abort はしない)
EXPECTED_REV="<git_rev>"
ACTUAL_REV="$(cd ../../.. && git rev-parse HEAD 2>/dev/null || echo unknown)"
if [ "$ACTUAL_REV" != "$EXPECTED_REV" ]; then
  echo "WARNING: code rev drift. Expected $EXPECTED_REV, got $ACTUAL_REV" >&2
  echo "         To fully reproduce, run: git checkout $EXPECTED_REV" >&2
fi
```

`git checkout <rev>` を **自動実行はしない** (user の作業状態を破壊しない安全側)。

## events.jsonl からの reproduce.sh 構築

events.jsonl の最初の `tool.bash.uv_run` イベントから:
- `cmd` フィールド (e.g., `"uv run python train.py --config config.yaml --seed 42"`)
- `git_rev` フィールド (e.g., `"a3f2c8d"`)
- `cwd` フィールド (e.g., `"/path/to/code"`)

→ これを `exec uv run --frozen python <relative entry path> --config config.yaml --seed 42` に変換。

複数 `uv run` 呼び出しがある場合 (e.g., preprocess + train) は **末尾の主実行のみ** 採用。
preprocess は config に hash として残っているはず (data_lineage.md)。

## uv.lock snapshot の取り方

Phase 6 で run が完了した直後、`research.experiment.run` skill が:

```bash
# 1. project root の uv.lock を 06_RUNS/<id>/uv.lock にコピー
cp uv.lock .research/<slug>/06_RUNS/<run_id>/uv.lock

# 2. uv.lock 不在時 (uv 未使用 project) は warning + skip
if [ ! -f uv.lock ]; then
  echo "WARNING: uv.lock not found, dependency snapshot skipped" >&2
  echo "         reproducibility checklist item #3 will be marked as missing" >&2
fi
```

uv.lock サイズは典型 100KB-1MB、複数 run で容量大きくなるが retention policy は v0.15+ で対応。

## Hardware info (events.jsonl.env)

PostToolUse hook (`.claude-plugin/hooks/post-experiment-log.sh`) で以下を記録 (既存 or 拡張):

```jsonl
{"event":"run.start", "env": {
  "os": "Linux 6.17.0-22-generic",
  "python": "3.11.9",
  "cuda_version": "12.4",
  "gpu_model": "NVIDIA A100-SXM4-40GB",
  "gpu_count": 1,
  "torch_version": "2.4.0+cu124"
}, "ts": "..."}
```

不足項目があれば lab.notebook が warning。

## Anti-patterns

- ❌ reproduce.sh から `set -euo pipefail` を省く (silent fail で再現性破壊)
- ❌ `uv run` を `uv run --frozen` 抜きで書く (deps drift で 1 ヶ月後動かない)
- ❌ uv.lock snapshot を skip (warning 無視)
- ❌ hardware info 不在で「同じ環境で動かしてください」とだけ書く
- ❌ data version hash を取らずに dataset を URL 参照のみ (URL は変わる、hash は不変)
- ❌ `git checkout <rev>` を reproduce.sh が自動実行 (user 作業破壊)

## 失敗 run 再現の意義

成功 run の再現性は当然として、**失敗 run の再現性こそ科学的価値**:
1. 同じ failure を別環境で再現できれば、failure mode が hardware / non-determinism でない確証
2. fix の効果検証 = "新 run で同 failure が起きないこと" の確認 (negative control)
3. 教育: 後続研究者が「この失敗パターン」を体験的に学べる
4. peer review: reviewer が失敗の再現性まで verify 可能

これが本 skill が **失敗 run も成功 run と同等の reproducibility 7-tuple を要求する理由**。
